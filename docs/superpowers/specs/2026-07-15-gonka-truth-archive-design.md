# 赛博记忆公墓 Gonka 真相档案增强设计

日期：2026-07-15

## 1. 目标

把当前功能型 MVP 收紧为一个明确契合 Gonka AI for Society 赛道的可验证数字遗产产品：针对消失的 Web2 社区，由两个 Gonka 模型独立核验公开证据，生成可追溯的 Truth Score、共识结论、数字墓志铭和永久档案。

本轮主案例只深挖虾米音乐。人人网和其他案例保留为展馆内容，不承担完整演示链路。

## 2. 范围

### 本轮实现

- 虾米音乐结构化策展证据包
- Wayback 历史快照补充
- Gonka 两个不同模型并行会诊
- 模型独立评分、分歧值、共识置信度和最终 Truth Score
- 真实 Request ID、模型名、请求时间和验证来源展示
- 真实会诊结果本地缓存
- 明确区分实时、缓存、部分成功和模拟结果
- SHA-256 内容哈希和不可变档案 JSON
- Pinata IPFS 上传与失败后的本地封存
- 基于档案哈希的纪念凭证和 NFT-ready metadata
- 一键演示状态机、超时、重试和进度反馈
- 服务端、前端核心逻辑和浏览器冒烟测试
- README、提交检查清单和运行说明

### 明确不实现

- 钱包连接
- 真实 NFT 或 SBT Mint
- Kite Agent Passport
- 真实付款
- Demo 视频录制
- 用户私有内容抓取
- 自动保存受版权保护的完整音乐、帖子或用户照片

## 3. 产品主线

完整演示流程：

1. 用户进入赛博记忆公墓并选择虾米音乐。
2. 系统加载策展证据包并尝试查询 Wayback 快照。
3. 两个 Gonka 模型分别扮演数字考古员和真相校验员。
4. 页面展示两个模型的独立评分、支持事实、不确定项、风险项和 Request ID。
5. 服务端计算分歧值、共识置信度和最终 Truth Score。
6. 系统根据可验证事实生成墓志铭，文学内容与事实字段分开展示。
7. 用户封存档案，系统冻结证据、模型输出、Request ID 和墓志铭并生成 SHA-256。
8. 配置 Pinata 时上传 IPFS，否则生成明确标记的本地封存档案。
9. 系统根据档案哈希生成可下载纪念凭证和 NFT-ready metadata。

## 4. 架构

保持现有技术栈：原生 HTML、CSS、JavaScript 前端加 Python `ThreadingHTTPServer` 服务。避免为黑客松新增重量级框架。

### 4.1 证据层

新增虾米音乐策展证据文件，保存：

- 证据 ID
- 主张内容
- 证据类型
- 来源标题
- 来源 URL
- 发布或快照日期
- 可信度级别
- 支持的事实字段
- 是否可公开归档
- 版权和隐私备注

策展证据是 Demo 的稳定底座。Wayback 数据只负责补充，不影响主流程可用性。

### 4.2 Gonka 会诊层

`POST /api/gonka/verify` 接收案例和证据摘要。服务端自动发现可用模型，从偏好池中选取两个不同模型并行请求。

每个模型规范化为：

```json
{
  "role": "digital_archaeologist",
  "model": "model-id",
  "requestId": "gonka-request-id",
  "requestedAt": "ISO-8601",
  "truthScore": 86,
  "summary": "结论摘要",
  "verifiedFacts": [],
  "uncertainClaims": [],
  "riskFlags": [],
  "fallback": false
}
```

聚合结果包含：

- `scoreSpread`：两个有效评分的绝对差
- `consensusConfidence`：基于评分接近度、有效模型数和证据完整度计算的 0 到 100 分
- `truthScore`：两个有效评分的均值，只有一个有效结果时使用该结果但降级状态
- `verificationState`：`live_consensus`、`partial`、`cached_live` 或 `demo_fallback`
- `sealEligibility`：`verified` 或 `draft`

### 4.3 缓存层

服务端将成功的双模型会诊结果写入 `tmp/cache/`。缓存键由案例 ID、证据包版本和模型组合共同生成。

缓存记录保存原始 Request ID、生成时间、模型和聚合结果。复用缓存时不得改写 Request ID，也不得显示为实时请求。

缓存只在以下情况使用：

- 两个实时模型都失败
- 用户主动选择使用最近一次真实会诊
- 一键演示在 API 超时后进行恢复

### 4.4 档案层

`POST /api/archive/seal` 冻结以下内容：

- 案例元数据
- 证据包版本和公开来源
- Gonka 原始规范化结果
- 聚合评分和验证状态
- 墓志铭
- 纪念册公开内容
- 生成时间和 schema 版本

服务端先对规范化 JSON 计算 SHA-256，再上传 Pinata。没有 `PINATA_JWT` 或上传失败时返回 `local-sealed`，不得生成伪 Arweave 地址。

档案内容使用首次生成时固定的 `contentCreatedAt`。上传重试时间只写入封存回执，不进入内容哈希。用户对同一份冻结内容重复封存时必须得到相同 SHA-256。

只有 `live_consensus` 和 `cached_live` 生成 `verified` 档案。`partial` 和 `demo_fallback` 只能生成 `draft` 档案。

### 4.5 纪念凭证层

纪念凭证由档案哈希确定性生成，包含案例名、Truth Score、验证状态、档案哈希、IPFS CID 或本地档案 ID、Gonka Request ID 和签发时间。签发时间沿用档案的 `contentCreatedAt`，不在每次下载时重新生成。

页面文案使用“生成纪念凭证”和“待链上认领”，不再显示已经完成 Mint。下载内容包括凭证 JSON 和 NFT-ready metadata JSON。

## 5. UI 设计

继续沿用 Digital Pompeii 的深色博物馆语言，不增加新的营销式页面。

### 墓碑页

- 保留墓志铭、遗物清单、时间线和纪念互动
- 新增“可验证事实”和“模型解释”视觉分区
- 将 NFT 按钮改为纪念凭证生成和下载

### 调查控制台

- 两位专家并排展示模型、评分、Request ID 和结论
- 增加分歧尺和共识置信度
- 顶部显示四种验证状态
- 对失败模型显示失败原因和单独重试入口

### 永久封存页

- 显示封存资格、内容哈希、存储提供方和档案状态
- IPFS 未配置时明确显示“本地封存，不具备永久存储证明”
- 增加重新上传入口和纪念凭证下载入口
- 删除 Kite 账单、额度和支付账本组件

### 一键演示

一键演示只控制页面流程，不录制视频。执行步骤为：选择虾米、运行证据探测、执行会诊、切换控制台、生成档案、展示纪念凭证。

每一步都有固定尺寸状态项，避免状态文字变化引发布局跳动。

## 6. 异常处理

### Wayback

- 成功：显示快照数量和时间范围
- 失败：继续使用策展证据，并标记 `curated_only`

### Gonka

- 两个模型成功：`live_consensus`
- 一个模型成功：`partial`
- 两个模型失败但存在真实缓存：`cached_live`
- 两个模型失败且无缓存：`demo_fallback`
- mock Request ID 必须以 `mock_` 开头并显示模拟标签

### IPFS

- 上传成功：返回 `ipfs://CID` 和可访问网关 URL
- 未配置：返回本地档案 ID 和配置提示
- 上传失败：保留本地档案和错误摘要，可重试上传

### 一键演示

- 每一步设置超时
- 失败时停止在当前步骤并显示重试
- 不吞掉错误，不自动把失败显示为成功
- API 恢复后可以从失败步骤继续

## 8. 验收标准

- 页面不再出现 Kite 支付和真实 NFT Mint 的误导性文案
- 虾米音乐公开证据可以逐条查看来源
- 两个 Gonka 模型分别显示真实模型名、Request ID 和评分
- 实时、缓存、部分成功和模拟状态可被清楚识别
- 双模型失败时主流程仍可恢复，但不会伪装成真实共识
- 同一份档案内容重复封存得到相同 SHA-256
- 配置 Pinata 后返回真实 CID；未配置时明确标记本地封存
- 纪念凭证和 NFT-ready metadata 可下载
- 一键演示可以从虾米案例完整走到封存结果
- 自动测试通过，服务可在 `http://127.0.0.1:5177/` 重复运行
